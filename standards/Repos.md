# Repos Convergence Standard — How `github-pr` Picks Targets

A structured standard that replaces global `gh search issues` flood with a **converged 4-tier pipeline**: 1–10 hand-curated core repos plus a network-derived auto-rotation, with the existing issue-level gates kept on top. Inspired by the followed-users digest pattern (`Repos.md` v0 article), but turned into machine-readable selection rules.

---

## Core Rule

> **Convergence over coverage.** Better to ship one PR a week to a repo we deeply care about than ten PRs a week to scattered 0-star projects. The pipeline must default to a small, high-trust target set and only reach further when that set is exhausted.

---

## Pipeline Overview

```
Tier 0  (curated, 1–10 repos)
   ∪
Tier 1  (network-derived, top N from following stream)
   │
   ├──── filtered by ────►  Tier 2  (BIG_ORGS exclusion)
   │
   ▼
For each repo in the converged set:
   gh search issues --repo=<r> label=help-wanted|good-first-issue|bug --limit=5
   │
   ▼
Tier 3  (per-issue gates: bot author, bot-noise label, ≥ MIN_REPO_STARS,
         AI policy, CLA, no duplicate PR on same SHA)
   │
   ▼
Cron picks 1–3 highest-value issues per tick.
```

The two failure modes this prevents:

1. **Tier 0 empty** — falls through to Tier 1 (network-derived) so the pipeline never stalls.
2. **Tier 0 + Tier 1 produce no claimable issue** — cron writes a SILENT report, no PR attempt.

## Tier 0 — Core 1–10 (manually curated)

Hand-maintained list, version-controlled at `Beatless/standards/repos.tier0.yaml`. Each entry MUST have:

```yaml
- repo: org/name           # required, unique
  rationale: |              # required, ≥ 1 sentence why this repo matters
    Why we believe contributing here pays off (used in commit msg + cron log).
  languages: [python, ts]   # required, repos are filtered by these langs
  contact_pattern: fast     # one of: fast (<24h response), normal (1-7d), slow (>7d)
  ai_policy: allowed        # one of: allowed | restricted | forbidden | unknown
  notes: |                  # optional, free-form (CLA, custom contribution flow, etc.)
    e.g. requires CLA, runs gating bot, etc.
```

**Selection criteria for Tier 0** (use to challenge any addition):

| Test | Pass condition |
|---|---|
| Stars | ≥ 200 (or under, with explicit rationale) |
| Activity | ≥ 1 merged PR in last 14 days |
| Maintainer responsiveness | "fast" or "normal" — no zombie repos |
| AI/LLM contribution policy | Not `forbidden` (see `Beatless/standards/PR.md` §AI-policy) |
| License | OSI-approved (MIT/Apache-2.0/BSD/MPL preferred) |
| Strategic fit | Solves a problem we touch in our own work (LLM infra, evals, agent harnesses, RL, BCI, observability) |

**Cap: 10 repos at any time.** If a new repo earns a slot, an existing one MUST be retired or moved to a `tier0_archive:` block in the same YAML.

## Tier 1 — Network-derived (auto-rotated)

Built nightly from the GitHub following stream. Rationale: the user already curated who to follow; that curation is the strongest possible signal. We mine each followee's recent star/fork/create activity to surface repos that already passed *their* taste filter.

### Build procedure

```
1. GET /users/{me}/following                    → followee list
2. For each followee:
       GET /users/{u}/events/public?per_page=30 → last events
3. From events, collect target repos by event type:
       CreateEvent  (refType: 'repository')   weight 5  (followee created it)
       ForkEvent                              weight 3  (followee forked it)
       WatchEvent   (i.e. star)               weight 2  (followee starred it)
       PullRequestEvent (action: opened)      weight 4  (followee is contributing)
4. Sum weights per repo → ranked candidate list.
5. Apply Tier 2 (BIG_ORGS) exclusion.
6. Apply Tier 0 dedup (Tier 0 wins ties).
7. Apply hard floors: ≥ MIN_REPO_STARS, has open issues with one of
   {good first issue, help wanted, bug}, license is OSI-approved.
8. Cap to top 8.
```

The output is `~/.hermes/state/repos.tier1.json`, regenerated by a dedicated cron at the start of each day. `github-pr.py` reads it; it does NOT recompute on every tick.

### Followee event API caps (matches Repos.md v0 article)

- 1 × `/following` request
- ≤ 100 × `/events/public` requests
- Total ≤ 131 requests/day, well under the 5000/h authenticated limit.

## Tier 2 — BIG_ORGS exclusion

```python
BIG_ORGS = {
    "anthropics", "anthropic-ai",
    "google", "google-deepmind", "google-gemini",
    "openai", "microsoft", "meta", "facebookresearch",
    "nvidia", "huggingface", "tensorflow", "pytorch",
    "MiniMax-AI", "Alibaba-NLP", "deepseek-ai"
}
```

A repo whose `org` (case-insensitive) is in `BIG_ORGS` is **automatically excluded from Tier 1** (we don't have leverage there). It MAY still appear in Tier 0 if explicitly added with a strong rationale (e.g. we've shipped there before and have a maintainer relationship).

## Tier 3 — Per-issue gates (existing, unchanged)

Already implemented in `~/.hermes/scripts/github-pr.py`:

1. **Bot author filter** — issue opened by a `[bot]`-suffixed account or a known bot login (`dependabot`, `renovate`, `codecov`, `pre-commit-ci`, etc.) → reject.
2. **Bot-noise label filter** — labels like `automated`, `dependencies`, `release`, `chore` → reject.
3. **Repo stars floor** — `< MIN_REPO_STARS` (default 20, env-overridable) → reject.
4. **AI/LLM policy scan** — fetch `CONTRIBUTING.md` / `.github/CONTRIBUTING.md`, look for AI/LLM-generated PR rules. If `forbidden`, reject. If `restricted`, mark `needs-human` and skip.
5. **CLA gate** — if a CLA is required and we haven't signed, mark `cla-blocked` and skip.
6. **Duplicate PR scan** — `gh pr list --search "#<issue-number>"` on the same repo → if any open or recently merged PR addresses the same issue, mark `duplicate` and skip.
7. **MISSING_PASSES guard** — pipeline must produce all 3 review pass scores; if any pass missing, the PR is auto-closed with apology comment (never bypassed).

## Discovery flow inside `github-pr.py`

The current `get_claimable_issues()` MUST be replaced with:

```python
def get_claimable_issues():
    targets = load_tier0() + load_tier1_state()       # ≤ 18 repos
    if not targets:
        return []
    issues = []
    for repo in targets:
        for label in ["good first issue", "help wanted", "bug"]:
            issues.extend(gh_search_issues_in_repo(repo, label, limit=5))
    issues = dedupe_by_repo_issue(issues)
    issues = preflight_filter(issues)                  # Tier 3
    issues.sort(key=lambda i: tier_score(i), reverse=True)
    return issues[:20]                                 # cap, cron picks top 1–3
```

Where `tier_score()` weights:

| Signal | Weight |
|---|---|
| Tier 0 membership | +100 |
| Tier 1 rank ≤ 3 | +30 |
| `good first issue` label | +20 |
| `help wanted` label | +15 |
| Tests already exist in repo | +10 |
| Repo stars ≥ 1000 | +5 |
| Issue updated in last 7 days | +5 |
| Issue body length 200–2000 chars (sweet spot) | +5 |

## When to update Tier 0

Trigger | Action
--- | ---
A Tier 0 repo merges 3 of our PRs without rejection | Promote contact_pattern → fast
A Tier 0 repo rejects 2 PRs in a row OR archives | Move to `tier0_archive`
A new repo gets 2+ ranked Tier 1 entries in 14 days | Eligible for Tier 0 promotion
Stars drop below 200 OR no merged PR in 30 days | Auto-flag for review (does not auto-demote)

5. Add the `chosen_tier` field to status reports so we can audit "are we contributing where we said we would?"

## Anti-patterns (do NOT do)

- ❌ `gh search issues --label='good first issue'` over the entire GitHub graph. That's the divergence we're escaping.
- ❌ Adding 50 repos to Tier 0 because "they look interesting". The cap is 10. Choose.
- ❌ Letting Tier 1 leak into Tier 0 implicitly. Every Tier 0 entry needs a written rationale.
- ❌ Skipping Tier 3 because "Tier 0 already vetted the repo". Issue-level filters protect against per-issue noise (bot triage, breaking changes by upstream).
- ❌ Targeting BIG_ORGS via Tier 1 just because a followee starred them. They have their own contribution funnel; we don't add value as a 50th opener.
